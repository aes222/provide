1. Project Description

The goal of this project is to develop a Fintech system for creating bank-verified digital identity, enabling users to authenticate with digital services without the need to upload identification documents such as ID cards or passports.

The system relies on the bank as a central trust anchor (Trust Anchor), which has already performed a complete legal and regulatory identification process of the user, and allows third parties to receive reliable and secure identity confirmation with the user's consent.

The system's vision is to enable every digital service to integrate a unified authentication button - "Identity Verification via Bank" - similar to the common use today of login buttons via Google or Facebook, but with a higher level of trust and in a manner that complies with financial regulation.

The project proposes a system for creating a verified digital identity layer that mediates between banks and digital services. The system enables banks to serve as identity providers, and external services to receive verified identity confirmation through simple and unified integration.

The system does not replace a bank KYC process, but uses it as a trust foundation. The user authenticates directly with their bank, and the bank confirms, with the user's consent, that this is a legal and verified identity. The digital service does not receive identification documents, but only a digital identity confirmation.

The system's vision is to create a new standard for digital authentication, where services will be able to display a unified authentication button of the platform, similar to existing global identity providers, while using the bank as a regulatory trust anchor.

2. Related Work

Today, digital services that require identity verification rely primarily on classic KYC processes, which include uploading identification documents, taking selfies, and manual or automatic verification. These processes are characterized by high friction, complex user experience, and significant risks related to storing sensitive documents and fraud.

In parallel, there are Open Banking / Open Finance solutions that allow access to the user's financial data, but these solutions do not serve as identity providers and do not provide official confirmation of the user's identity to a third party. Banks, despite being the entity that performed the strictest identification, do not currently serve as digital identity providers for external services.

As a result, many organizations continue to require uploading physical documents, even when it would be possible to rely on an identity that has already been verified at the banking level.

Current Solutions & Limitations

  • Traditional KYC: High friction, complex UX, security risks from document storage
  • Open Banking/Finance: Provides financial data access but not identity verification services
  • Social Login (Google/Facebook): Convenient but lacks regulatory compliance for financial services
  • Bank Identity Gap: Banks perform strictest verification but don't serve as identity providers

3. Functional Description / Requirements

Basic Functional Description

A digital service integrates an authentication button of the platform - "Identity Verification via Bank" - into the user interface. After clicking the button, the user is redirected to an authentication process with their bank through a secure redirect.

During authentication, the bank performs strong verification using existing means (password, OTP, biometrics), and after user approval, returns a digital identity confirmation (Identity Assertion) to the system. This confirmation includes verified identity details as needed, such as name match, ID number, age, or country, without transferring documents or financial data.

The system provides the digital service with a reliable indication of the user's identity, which can be used one-time or continuously, according to the service's policy.

Core Use Cases

Loan & Financial Services

Complete replacement of uploading identification documents and selfie verification in a bank-verified authentication process.

E-commerce & Trading Platforms

Identity verification for customers and sellers, opening verified accounts, and reducing identity fraud.

Sensitive Digital Services

Insurance, crypto, age-based or citizenship-based services, and highly regulated services.

Functional Requirements

  • Secure OAuth 2.0 Authorization Code Flow for bank authentication
  • Identity assertion generation and validation
  • Token management and lifecycle handling
  • Audit logging for compliance and security
  • RTL (Right-to-Left) UI support for Hebrew interface
  • Responsive design for mobile and desktop devices
  • No storage of identification documents or bank login credentials
  • Support for multiple banks through authorized Open Banking intermediaries
  • Configurable identity attributes (name, ID, age, country) based on service needs
  • One-time or continuous identity verification modes

4.1. Each Module Description

Authentication Module

Handles OAuth 2.0 flow initiation, redirect management, and token exchange. Manages secure communication with bank authentication endpoints through authorized Open Banking intermediaries.

Identity Assertion Service

Generates and validates digital identity assertions based on bank-verified user data. Manages identity attribute mapping and ensures compliance with data privacy regulations.

Token Management Module

Manages OAuth tokens, refresh tokens, and session management. Implements secure token storage, expiration handling, and revocation mechanisms.

API Gateway

Provides RESTful API endpoints for digital services to integrate identity verification. Handles request validation, rate limiting, and API versioning.

Audit & Compliance Module

Logs all identity verification events, maintains audit trails for regulatory compliance, and implements data retention policies. Ensures GDPR and financial regulation compliance.

Client UI Module

Provides authentication button components, redirect handling, and user feedback. Supports RTL layout for Hebrew interface and responsive design for all devices.

6.1. Usage Illustration

Scenario 1: Loan Application

  1. User visits a loan service website and clicks "Apply for Loan"
  2. Service displays "Verify Identity via Bank" button
  3. User clicks the button and is redirected to their bank's authentication page
  4. User authenticates with bank using existing credentials (password, OTP, biometrics)
  5. Bank confirms identity and redirects back to loan service
  6. Loan service receives verified identity assertion and proceeds with application
  7. No document upload required - identity is verified instantly

Scenario 2: E-commerce Account Creation

  1. User wants to create a seller account on an e-commerce platform
  2. Platform requires identity verification for sellers
  3. User clicks "Verify via Bank" authentication button
  4. After bank authentication, platform receives verified identity
  5. Account is created instantly with verified status
  6. User can start selling immediately without document verification delays

Scenario 3: Age-Restricted Service

  1. User attempts to access an age-restricted service (e.g., crypto trading)
  2. Service requires age verification
  3. User authenticates via bank using ProvIDe
  4. Service receives age confirmation (without full ID details)
  5. Access is granted based on verified age attribute
  6. Privacy is maintained - only necessary information is shared

7. Server Side

The server-side platform serves as the core identity verification service, managing authentication flows, identity assertions, and API endpoints for digital services. Built with modern backend technologies (.NET, Node.js, or Python), the server ensures security, scalability, and regulatory compliance.

Server Architecture

  • RESTful API architecture for service integration
  • Microservices-based design for scalability
  • Secure token management and storage
  • Identity assertion generation and validation
  • Integration with Open Banking intermediaries
  • Comprehensive audit logging system
  • Data encryption at rest and in transit
  • Rate limiting and API security measures

7.1. API

Authentication Endpoints

POST /api/v1/auth/initiate

Initiate OAuth 2.0 authentication flow

POST /api/v1/auth/callback

Handle OAuth callback from bank

POST /api/v1/auth/token

Exchange authorization code for tokens

Identity Verification Endpoints

POST /api/v1/identity/verify

Request identity verification for a user

GET /api/v1/identity/assertion/:id

Retrieve identity assertion by ID

GET /api/v1/identity/status/:sessionId

Check verification status

Service Management Endpoints

POST /api/v1/services/register

Register a new digital service

GET /api/v1/services/:serviceId/config

Get service configuration and API keys

Audit & Compliance Endpoints

GET /api/v1/audit/logs

Retrieve audit logs (admin only)

GET /api/v1/compliance/report

Generate compliance reports

API Authentication

All API endpoints require authentication using API keys or OAuth 2.0 client credentials. Services must register and obtain API credentials before accessing the platform.
